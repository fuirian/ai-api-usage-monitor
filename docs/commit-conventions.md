# 커밋 규칙

커밋 메시지 형식과 작성 습관의 **정본**이다. 브랜치·병합 흐름은 `docs/branch-conventions.md`를 따른다.

---

## 1. 기본 원칙

- **1 Action 1 Commit**에 가깝게 유지한다. 하나의 커밋에 서로 무관한 변경을 섞지 않는다.
- 메시지가 길어지면 **커밋을 나누는 것**을 우선한다.

---

## 2. 메시지 구조

커밋 메시지는 **제목 · 본문 · 꼬리말** 세 부분으로 나누고, **각 부분 사이에 빈 줄**로 구분한다.

```
Type: Subject

Body

Footer
```

- **`Type: Subject`** 한 줄이 제목 역할을 한다. (`type : subject` 형태로 통일해도 되나, 팀에서는 아래 **Type 표기**를 따른다.)

---

## 3. 제목 (Type + Subject)

- **Type(태그)** 는 **영어**로 쓰고 **첫 글자는 대문자**로 한다.
- **Subject(제목)** 는 **최대 약 50자**를 넘지 않게 짧게 쓴다.
- 제목 끝에 **마침표·불필요한 특수문자**는 쓰지 않는다.
- Subject를 **영어**로 쓸 때는 **동사 원형을 앞에** 두고, 첫 글자는 대문자로 쓴다. (팀에서 한글 제목을 쓰는 경우도 가능하나, 형식은 통일한다.)

### 3.1 Type 의미와 예시

| Type | 의미 | 예시 |
|------|------|------|
| Feat | 새 기능 | `Feat: OpenAI 프록시 호출 기능 구현` |
| Fix | 버그 수정 | `Fix: 토큰 계산 로직 오류 수정` |
| Docs | 문서만 변경 | `Docs: README 팀원 정보 업데이트` |
| Style | 포맷·세미콜론 등 비기능 스타일 | `Style: 오타 수정 및 들여쓰기 조정` |
| Refactor | 동작 유지 리팩터 | `Refactor: 중복 인증 로직 공통화` |
| Test | 테스트 추가·수정 | `Test: 프록시 컨트롤러 단위 테스트 추가` |
| Chore | 빌드·의존성·도구 등 | `Chore: build.gradle 의존성 추가` |

---

## 4. 본문 (Body)

- 한 줄당 **약 72자 이내**를 권장한다.
- **어떻게**보다 **무엇을·왜** 바꿨는지 위주로 쓴다.

---

## 5. 꼬리말 (Footer, 선택)

- 이슈 트래커와 연결할 때 사용한다.
- 형식: **`유형: #이슈번호`**  
  여러 이슈는 **쉼표(,)** 로 구분한다.

| 유형 | 용도 |
|------|------|
| Fixes | 해당 이슈를 **수정 중**(아직 완전 해결 아님) |
| Resolves | 해당 이슈를 **해결 완료**했을 때 |
| Ref | 참고 이슈 |
| Related to | 관련은 있으나 아직 해결되지 않은 이슈 |

예:

```text
Fixes: #23 Related to: #11, #12
```

---

## 6. 문서 유지

- 타입 목록이나 제목 규칙이 바뀌면 본 문서를 먼저 수정한다.
